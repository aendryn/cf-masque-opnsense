# Cloudflare Zero Trust OPNsense Plugin — Architecture Design

## Overview

An OPNsense plugin integrating Cloudflare Zero Trust networking as a first-class VPN component. The plugin supports every Cloudflare Zero Trust transport protocol and integrates fully with OPNsense's MVC framework, configd, routing, DNS, and firewall subsystems.

---

## Protocol Support

### WARP via MASQUE (Primary)
- **Protocol**: Connect-IP (RFC 9484) over HTTP/3 + QUIC
- **Fallback**: HTTP/2 CONNECT-IP over TLS
- **Authentication**: ECDSA P-256 key pair, enrolled via Cloudflare WARP API
- **Endpoint**: `162.159.198.1:2408` (UDP/QUIC), `162.159.198.2:443` (TCP/HTTP2)
- **SNI**: `consumer-masque.cloudflareclient.com`
- **Implemented by**: `cfzt-warp` Go daemon (see daemon section)

### WARP via WireGuard
- **Protocol**: WireGuard UDP
- **Authentication**: Curve25519 key pair, registered via WARP API
- **Peer**: Configured from API registration response
- **Implemented by**: FreeBSD kernel WireGuard (built-in FreeBSD 14+, wireguard-kmod for older)

### Cloudflare Tunnel (cloudflared)
- **Protocol**: Proprietary Cloudflare Tunnel protocol (QUIC/HTTP2)
- **Authentication**: Tunnel token or certificate
- **Implemented by**: Official `cloudflared` daemon

### Future Protocols
The `IProtocolDriver` interface abstracts protocol backends, enabling new protocols without plugin rewrites.

## API

### Cloudflare WARP Registration API
- Base URL: `https://api.cloudflareclient.com`
- Version: `v0a4471` (discovered via usque reverse engineering)
- Headers: `User-Agent: WARP for Android`, `CF-Client-Version: a-6.35-4471`

**Endpoints used**:
- `POST /v0a4471/reg` — Register device (WireGuard key initially)
- `PATCH /v0a4471/reg/{id}` — Enroll MASQUE key
- `GET /v0a4471/reg/{id}/account` — Get account info
- `PUT /v0a4471/reg/{id}/account` — Update license key
- `GET /v0a4471/reg/{id}/account/devices` — List devices

**Registration flow** (MASQUE mode):
1. POST to `/reg` with WireGuard pubkey → receive device ID + access token
2. Generate ECDSA P-256 key pair
3. PATCH to `/reg/{id}` with MASQUE pubkey (DER-encoded, base64) → get assigned IPs + peer endpoint
4. Establish MASQUE connection using TLS client cert (derived from ECDSA key)

### Cloudflare Zero Trust API (Dashboard API)
- Base URL: `https://api.cloudflare.com/client/v4`
- Auth: Bearer token (Cloudflare API token with Zero Trust permissions)
- Used for: tunnel management, Gateway policies, Access integration, DNS policies

## `cfzt-warp` Go Daemon

Handles WARP/MASQUE protocol. Runs as a daemon on FreeBSD.

**Source**: `src/usr/local/go/cfzt-warp/`
**Installed**: `/usr/local/sbin/cfzt-warp`

### Key components:
- **`api/`** — Cloudflare WARP API client (register, enroll, account management)
- **`masque/`** — MASQUE/QUIC tunnel establishment and maintenance
- **`daemon/`** — Daemon lifecycle (PID file, signals, syslog, config reload)
- **`config/`** — JSON config loading (generated by OPNsense configd templates)
- **`tun/`** — FreeBSD TUN device management (via `songgao/water`)

### FreeBSD TUN support
Uses `github.com/songgao/water` which supports BSD `/dev/tun` devices. The TUN interface is pre-created by the OPNsense plugin (using `ifconfig tun` or the `if_tun` kernel module).

### Dependencies:
- `github.com/quic-go/quic-go` — QUIC/HTTP3
- `github.com/Diniboy1123/connect-ip-go` — Connect-IP (MASQUE) protocol
- `github.com/songgao/water` — Cross-platform TUN/TAP

## Config Model (CloudflareZT.xml)

### Storage: `//OPNsense/CloudflareZT` in config.xml

Non-sensitive fields stored in config.xml:
- Organization name, account_id, team_name
- Connection settings (protocol, IPs, endpoints, MTU, split tunnel rules)
- Monitoring settings
- DNS settings

### Secrets storage
Sensitive values (API tokens, private keys, device tokens) stored in:
- `/usr/local/etc/cloudflarezt/secrets.json`
- Encrypted with AES-256-GCM
- Key derived from `/etc/machine-id` + plugin-specific salt
- Mode 0600, root:wheel only
- Referenced from config.xml by `{type}:{uuid}` reference keys

## Backend Service Architecture

### Service processes
1. **`cfzt-warp`** — Go daemon, one instance per WARP connection
   - Config: `/usr/local/etc/cloudflarezt/connections/{uuid}.json`
   - PID: `/var/run/cfzt-warp-{uuid}.pid`
   - Log: syslog facility `daemon`, ident `cfzt-warp[{name}]`

2. **`cloudflared`** — Official Cloudflare daemon, one per tunnel
   - Config: `/usr/local/etc/cloudflarezt/tunnels/{uuid}.yml`
   - PID: `/var/run/cloudflared-{uuid}.pid`

3. **`cfzt-monitor`** — Python health monitoring daemon
   - Monitors all active connections
   - Posts health metrics to OPNsense notification system
   - Triggers reconnect on failure
   - PID: `/var/run/cfzt-monitor.pid`

### configd actions (actions_cloudflarezt.conf)
- `cloudflarezt start` — Start all enabled connections
- `cloudflarezt stop` — Stop all connections
- `cloudflarezt restart` — Restart all
- `cloudflarezt startconn %s` — Start specific connection by UUID
- `cloudflarezt stopconn %s` — Stop specific connection
- `cloudflarezt status` — Overall status
- `cloudflarezt connstatus %s` — Status for specific connection
- `cloudflarezt register %s` — Register device for connection
- `cloudflarezt enroll %s` — Re-enroll key for connection
- `cloudflarezt rotatekey %s` — Rotate key for connection
- `cloudflarezt diagnostics %s` — Run diagnostics
- `cloudflarezt logs %s` — Fetch recent logs
- `cloudflarezt template reload OPNsense/CloudflareZT` — Reload templates

## Firewall & Routing Integration

### Interfaces
- Each WARP connection creates a TUN interface: `cfzt0`, `cfzt1`, etc.
- cloudflared tunnels use virtual routing (no host interface needed)
- Interfaces registered with OPNsense via `plugins.inc.d` `_devices()` hook

### Routing
- Split tunnel: routes for included prefixes added to TUN interface
- Full tunnel: default route (0.0.0.0/0) via TUN, except management traffic
- Multi-WAN: uses OPNsense gateway groups, cfzt-warp binds to specific WAN

### Firewall aliases
- Per-connection alias `cfzt_{name}_peers` containing Cloudflare endpoint IPs
- Per-connection alias `cfzt_{name}_routes` containing routed prefixes

## Reliability Design

### Reconnection
- Exponential backoff: 1s, 2s, 4s, 8s, max 60s
- Context-aware: respects shutdown signals
- WAN change detection via `route` socket monitoring

### Key rotation
- Automated via cron (configurable interval, default 30 days)
- Generates new ECDSA key pair
- Enrolls via PATCH API
- Graceful handover (old connection stays up during re-enrollment)

### WAN failover
- cfzt-warp listens on routing socket for gateway changes
- On WAN change: rebind UDP socket, reconnect QUIC session
- cloudflared handles failover internally

## Security Design

### Principle of least privilege
- cfzt-warp runs as `nobody` after creating TUN interface (root required for tun creation)
- Or: use `setuid` helper for TUN creation, then drop to `nobody`
- API token stored encrypted, decrypted only when needed
- No secrets in config.xml

### Input validation
- All API inputs validated via OPNsense model field types
- PHP controllers use `ApiMutableModelControllerBase` which validates model fields
- Python scripts sanitize all inputs before passing to subprocesses (no shell=True)
- Go daemon uses typed config struct, no string interpolation into commands

### Shell injection prevention
- All subprocess calls use argument arrays (never string interpolation)
- Go: `exec.Command("cmd", arg1, arg2)` — never `exec.Command("sh", "-c", ...)`
- Python: `subprocess.run(["cmd", arg1], ...)` — never `shell=True`
